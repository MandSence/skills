# 基于设计文档生成代码

本技能接收系统设计说明文档（Markdown格式）并生成完整的、生产就绪的源代码。

## 文档发现与输出

**文档命名**：`[系统名称][版本号]_系统设计说明文档.md`

**发现顺序**：
1. 优先：用户提供的具体路径
2. 其次：当前工作目录下搜索
3. 最后：递归搜索所有子目录（指定最大深度为3层）

**输出目录**：`[系统名称]_源代码/`

**异常处理**：
- 文档未找到 → 提示用户提供路径
- 内容不完整/矛盾 → 列出问题询问用户
- Mermaid解析失败 → 尝试自动修复，修复失败时说明问题并询问是否跳过

**Mermaid 语法错误处理流程**：
1. 尝试自动修复常见语法错误（如保留字作为节点ID：`end` → `finish`，`graph` → `diagram`）
2. 尝试自动修复常见结构问题（如连接符错误、节点形状错误）
3. 无法修复时，向用户报告具体错误位置和原因
4. 提供以下选项给用户：手动修复图表、跳过该图表、简化图表

## 工作流程

### 步骤1：分析设计文档

提取系统名称、功能需求、数据实体、API端点、UI组件、业务逻辑。

**Mermaid图表映射**（详细请参考 `references/mermaid-code-mapping-reference.md`）：

| 类型 | 元素 | 代码映射 |
|------|------|----------|
| 流程图 | 决策 `{}` | if/else, switch/case |
| 流程图 | 并行分支 | Promise.all(), 多线程 |
| 流程图 | 循环箭头 | for/while, 重试逻辑 |
| 时序图 | actor | 类/模块/微服务 |
| 时序图 | `->` | 同步方法调用 |
| 时序图 | `-->` | 异步消息/消息队列 |
| 时序图 | alt/opt/loop | 条件/可选/循环结构 |
| 类图 | class | class/interface |
| 类图 | `--|>` | extends |
| 类图 | `..|>` | implements |
| 类图 | 组合 `*-->` | 强关联关系 |
| 状态图 | 状态 | enum |
| 状态图 | 状态转换 | 状态机类 + transition 方法 |
| 状态图 | 入口/退出 | onEnter()/onExit() |
| ER图 | 实体 | @Entity |
| ER图 | PK/FK | @Id / @ForeignKey |
| ER图 | `||o`/`o||` | @OneToOne |
| ER图 | `}o`/`o{` | @OneToMany |
| ER图 | `}o{`/`o{o` | @ManyToMany |


**输出功能分析摘要**（步骤6验证时对照使用）：

```markdown
## 功能分析摘要

### 核心功能需求
- [ ] 功能1：描述 → 位置（章节/图表）
- [ ] 功能2：描述 → 位置
...

### Mermaid图表清单
| 图表类型 | 描述 | 关键节点/转换 |
|---------|------|---------------|
| 流程图 | 订单处理流程 | A→B→C(条件) |
| 状态图 | 订单状态机 | PENDING→PAID→SHIPPED |
...

### 数据实体
- [ ] 实体1：字段、关系
- [ ] 实体2：字段、关系
...

### API端点
- [ ] GET /api/users
- [ ] POST /api/orders
...

### UI页面
- [ ] 登录页
- [ ] 订单列表页
...
```

### 步骤2：设计系统架构

**优先使用设计文档中的架构设计**：

1. **检查设计文档**：确认文档是否明确描述了以下内容：
   - 系统架构模式、技术栈（后端/前端/AI框架）、数据库、缓存、消息队列等

2. **架构设计决策**：

   - **情况A：文档已包含架构设计** → 直接按照文档实施，跳过2.1-2.4，进入步骤3
   - **情况B：文档未包含或部分包含** → 按2.1-2.4进行设计，缺失项需要补充

3. **情况A：按文档设计实施**
   - 直接采用文档中的架构设计和技术栈
   - 如发现明显不合理的技术选型（如语言框架不匹配），询问用户
   - 按文档中的目录结构创建项目

4. **情况B：补充架构设计**（参考 `references/architecture-and-tech-stack-reference.md`）

   #### 2.1 评估规模
   - 小型 → 分层架构(MVC) | 中大型 → 微服务 | 高并发 → 事件驱动

   #### 2.2 选择技术栈
   - 小型：Python/FastAPI 或 Node.js/Express + Vue3
   - 中型：Java/Spring Boot + Vue3
   - 大型/高并发：Go/Gin 或 Java + Spring Cloud + React

   #### 2.3 AI能力对接评估（如包含AI功能）
   - Python：LangChain / LlamaIndex / Haystack
   - Java：LangChain4j / Spring AI / Semantic Kernel
   - 详细选型参考 `architecture-and-tech-stack-reference.md` 章节4

   #### 2.4 确定目录结构
   - AI项目采用 `4.4 AI项目目录结构`，其他项目采用标准目录结构

5. **用户确认**（情况B需要）：展示架构设计后确认，进入步骤3

### 步骤3：并行生成代码（三阶段）

| 阶段 | 后端任务 | 前端任务 | 并行方式 |
|------|---------|---------|----------|
| **阶段1** | 项目设置、数据库层、模型、验证 | 项目设置、路由、状态管理、基础UI组件 | 完全并行 |
| **阶段2** | API契约定义 + 业务逻辑 | API集成（基于Mock契约）+ 页面视图 | 先定义契约，前后端基于Mock并行开发 |
| **阶段3** | 中间件、安全 | 状态管理、样式 | 完全并行 |

**Mermaid驱动生成原则**：
- 对于每个Mermaid图表，生成时显式说明"实现图表中的X节点"
- 确保流程图的所有决策分支都有对应代码
- 确保状态图的所有状态转换都有实现
- 确保时序图的所有参与者、消息都有对应模块/方法
- 确保ER图的所有实体、关系都已创建

**AI代码生成原则**（适用于包含AI功能的项目）：
- **RAG实现**：必须包含文档加载、文本分割、向量嵌入、检索器、问答链完整流程
- **Agent实现**：必须包含工具定义、工具绑定、Agent创建、执行循环
- **流式响应**：对于聊天场景，必须支持流式输出（Streaming）
- **记忆管理**：对话类应用必须实现记忆（Memory）功能，支持消息历史存储和检索
- **提示词管理**：将系统提示词、用户提示词模板分离存储，便于维护和版本管理
- **成本追踪**：记录每次AI调用的Token消耗和成本
- **错误处理**：AI调用失败时要有降级策略和重试机制
- **响应缓存**：对相同问题的回答进行缓存，减少AI调用成本

**生成原则**：

1. **必须应用代码增强规范**（详见"代码增强规范"章节）：
   - 后端：多级缓存、分页排序、参数验证、事务管理、限流熔断
   - 前端：防抖节流、乐观更新、请求重试、缓存策略

2. **Mermaid图表驱动实现**：
   - 状态图 → 完整状态机实现（含历史记录、钩子函数、转换验证）
   - 流程图决策 → 策略模式/规则引擎/条件分支
   - 时序图循环 → 批量处理 + 并发控制
   - 时序图虚线（异步）→ 消息队列/事件发布订阅
   - 类图继承/实现 → 使用 extends / implements + 抽象类/接口

3. **算法场景识别与实现**：
   - 大数据量搜索（>1000条 或 超过单页显示数量） → 二分查找/布隆过滤器
   - 列表展示 → 游标分页（无限滚动）/ 偏移分页（传统分页）
   - 并发操作（>10个 或 用户同时操作） → 信号量/互斥锁/令牌桶
   - 热点数据（高频访问 或 缓存命中率 > 80%） → LRU/LFU缓存
   - 批量查询优化 → Map聚合（解决N+1查询）
   - 数据去重 → Set/布隆过滤器
   - 搜索/输入框 → 防抖（debounce，300-500ms）
   - 滚动事件 → 节流（throttle，100-200ms）

4. **代码行数与质量双重保障**：
   - 每个Service类包含至少2-3个增强方法（如缓存、验证、日志）
   - 每个API端点包含完整的错误处理和日志记录
   - 每个Repository方法包含分页和排序支持
   - 每个前端组件包含加载状态和错误处理

### 步骤4：代码质量检查

检查语法、结构，修复问题。无法修复时说明问题并寻求指导。

**代码质量检查清单**：

| 检查类别 | 核心检查项 |
|---------|-----------|
| **后端-数据层** | 分页+排序支持、批量操作接口 |
| **后端-业务层** | 参数验证、状态机实现、事务管理 |
| **后端-API层** | 限流保护、熔断降级、请求日志（脱敏）、错误统一处理 |
| **后端-安全** | 输入验证、SQL注入防护、XSS防护、敏感数据加密 |
| **前端-数据获取** | 防抖/节流、请求取消、乐观更新 |
| **AI项目** | RAG（多路召回+重排序）、Agent（工具定义+重试）、对话（记忆+流式输出）、通用（提示词管理+Token追踪+限流+熔断+重试+缓存） |


### 步骤5：设计文档合规性审查

对照步骤1的**功能分析摘要**逐项验证：

```markdown
验证检查清单：
□ 所有核心功能需求是否已实现？
□ 所有Mermaid图表的关键节点是否已实现？
□ 所有数据实体是否已创建？
□ 所有API端点是否已创建？
□ 所有UI页面是否已实现？
```

不匹配时列出不匹配项并询问用户。

### 步骤6：代码行数检查与扩展

**行数要求**：后端代码中有效代码行数（排除代码注释、空行）≥ 3000 行

**代码行数不满足时的扩展策略**：
如果检测到后端代码有效代码行数不满足 3000 行要求，按以下优先级扩展代码：

1. **核心功能扩展**：围绕系统核心功能增加辅助方法和逻辑
   - 为核心 Service 类添加更多辅助方法（如导出、统计、批量操作）
   - 为核心 API 端点添加更多查询条件和过滤选项
   - 为核心数据实体添加更多关联查询方法

2. **补充辅助功能代码**：
   - 验证工具类（数据格式验证、业务规则验证）
   - 业务辅助函数（日期处理、字符串处理、加密解密）
   - 日志服务（结构化日志、日志分级、日志脱敏）
   - 异常处理器（全局异常捕获、自定义异常、错误码映射）
   - 批量操作工具（批量插入、批量更新、批量删除）
   - 安全工具（密码哈希、JWT验证、签名验证）
   - 限流器（基于IP、基于用户、滑动窗口算法）
   - 熔断器（Hystrix模式、半开启状态）

3. **文档注释扩展**：
   - 为所有公共接口添加详细的文档注释
   - 为复杂算法添加实现说明


**补充策略**（代码行数不足时按优先级添加）：

- **后端**：验证工具类、业务辅助函数、日志服务、异常处理器、批量操作工具、安全工具、限流器、熔断器
- **前端**：UI组件库、状态管理扩展、表单验证hooks、骨架屏组件、错误边界组件、防抖/节流hooks、虚拟滚动组件
- **AI项目**：提示词管理器、多路召回器、重排序器、文档处理器、工具注册器、会话管理器、Token计数器、AI调用追踪器、响应缓存器、流式处理器、降级处理器、敏感信息检测器

## 指导原则

1. **丰富逻辑**：错误处理、验证、缓存
2. **算法集成**：二分搜索、排序、缓存算法等
3. **设计一致**：功能、数据结构、API、UI匹配文档
