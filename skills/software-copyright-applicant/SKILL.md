# 软件著作权申请材料生成技能

本技能专用于生成软件著作权申请所需的各种材料，包括系统设计说明书、源代码、源代码提取文件等。

## 路径约定

- `<cwd>`：表示用户当前工作目录，例如 `/Users/wanghao/Desktop/self_cust/test_one`

## 一、工作流程

当用户需要生成软著申请材料时，按照以下流程操作：

### 1. 理解需求与架构设计

#### 1.1 前置询问：核心功能分析内容

在开始工作前，**必须**先询问用户是否能提供软著对应系统的核心功能分析内容：

- 使用 `AskUserQuestion` 工具询问用户：
  - 问题：是否已准备好该软件系统的核心功能分析内容？
  - 选项：
    - "是，我已准备好核心功能分析内容" - 用户将提供详细的核心功能分析
    - "否，请根据软件名称自动分析" - 技能将根据软件名称自动推断核心功能

- **如果用户选择"是"**：
  - 让用户提供核心功能分析内容（可以是文字描述、功能列表、业务流程等）
  - 记录用户提供的内容，在后续生成系统设计说明书时严格按照用户提供的内容规划核心功能

- **如果用户选择"否"**：
  - 在后续步骤中根据用户提供的软件名称自动推断该系统可能包括的核心功能

#### 1.2 收集基础信息并设计架构

根据前置询问的结果，进行系统架构设计和技术栈选择：

- **如果用户已提供系统核心功能分析内容**：
  - 基于用户提供的内容进行系统架构的设计
  - 根据业务需求进行技术栈的选择（包括前端、后端、数据库等）

- **如果用户未提供系统核心功能分析内容**：
  - 基于用户提供的软件名称自动推断该系统可能包括的核心功能
  - 尽可能贴合实际应用、设计有意义的功能模块
  - 进行系统核心功能的规划、系统架构的设计、技术栈的选择

### 2. 生成Markdown格式的系统设计说明书

**重要**：这是一个复杂任务，请严格按照以下分步骤执行：

#### 步骤2.1：创建任务清单

使用 `TodoWrite` 工具创建以下任务清单：
- 读取系统设计说明书模板文件
- 生成引言部分（一、引言）
- 生成软件总体设计部分（二、软件总体设计）
- 生成详细功能设计部分（三、详细功能设计）
- 生成技术实现说明部分（四、技术实现说明）
- 生成运行测试说明部分（五、运行测试说明）
- 生成软件功能列表部分（六、软件功能列表）
- 生成数据库设计部分（七、数据库设计）
- 生成接口文档部分（八、接口文档）
- 创建 assets 目录
- 保存完整的 Markdown 文件

#### 步骤2.2：读取模板文件

使用 `Read` 工具读取模板文件：
`./references/system-design-template.md`

#### 步骤2.3：逐章节生成内容

 **立即开始生成**：请立即开始按照下述顺序逐章节生成内容。

执行顺序（每完成一个章节立即标记对应 todo 为完成，然后继续下一个）：

**重要**：所有 Mermaid 图表代码直接作为 Markdown 代码块（\`\`\`mermaid ... \`\`\`）写入文档内容中，在后续步骤4中会自动转换为 PNG 图片。

1. **一、引言**：包含软件名称、系统概述（系统概述、开发目标）、开发时间、运行环境、目标读者、参考资料（罗列1-3项）
2. **二、软件总体设计**：架构设计（含 Mermaid 架构图代码块）、核心功能概述
3. **三、详细功能设计**：各模块功能描述、业务流程图（含 Mermaid 流程图代码块）、输入输出
4. **四、技术实现说明**：核心技术选型（后端、前端技术栈）、关键技术点实现逻辑
5. **五、运行测试说明**：测试环境、测试用例
6. **六、软件功能列表**
7. **七、数据库设计**：表结构设计（详细的表结构定义）、实体关系图（使用 Mermaid erDiagram 代码块）
8. **八、接口文档**：接口列表（按模块组织的RESTful API文档，包含路径、描述、参数、响应示例）、接口调用示例（展示完整的请求和响应示例）、通用错误码说明

#### 步骤2.4：保存文件

使用 `Write` 工具将生成的完整内容保存为 Markdown 文件，文件名格式：`<cwd>/[软件名称]_系统设计说明书.md`

#### 步骤2.5：创建资源目录

使用 `Bash` 工具创建 `<cwd>/assets` 目录（用于存放后续生成的 PNG 图片）

#### 步骤2.6：任务完成确认

标记所有 todo 为已完成状态

### 3. 确认Markdown格式的系统设计说明书

在生成 Markdown 格式的系统设计说明书后，**必须**先与用户确认是否需要进行调整：

- 使用 `AskUserQuestion` 工具询问用户：
  - 问题：系统设计说明书已生成，是否需要进行调整？
  - 选项：
    - "不需要调整，继续生成Word文档" - 用户确认内容满意，继续后续流程
    - "需要调整" - 用户需要修改文档内容

- **如果用户选择"不需要调整"**：
  - 直接进入第4步，生成Word格式系统设计说明书

- **如果用户选择"需要调整"**：
  - 询问用户具体需要调整的内容和修改要求
  - 根据用户的要求修改系统设计说明书
  - 修改完成后，再次询问用户是否满意
  - 重复此步骤，直到用户确认不需要调整为止
  - 只有在用户确认后，才进入第4步

- **重要**：用户确认后，生成的Word文档将基于当前Markdown文件内容，请确保Markdown文件是最终确认版本

### 4. 生成Word格式的系统设计说明书

在用户确认后，**基于最新的Markdown格式的系统设计说明书文件（即最终确认版本）**，按以下顺序生成Word文档 (.docx格式) ：

1. **调用脚本转换Mermaid图表为PNG**
   - 使用 `./scripts/convert_mermaid_to_png.py` 转换文档中的Mermaid代码块
   - PNG图片保存到 `<cwd>/assets/` 目录
   - 图片命名格式：`[软件名称]_图表[序号].png`
   - 支持重试机制，确保所有图表成功转换

2. **调用脚本生成.docx文档**
   - **确保传入的Markdown文件是用户确认后的最新版本**
   - 使用 `./scripts/generate_docx.py` 生成Word文档
   - 自动生成文档目录（基于一级、二级、三级标题）
   - 将Mermaid代码块替换为PNG图片
   - 应用标准样式（宋体正文、黑体标题、1.5倍行距）
   - 标题颜色统一为黑色
   - 去除Markdown格式标记（####、**等）
   - 文件名格式：`[软件名称]_系统设计说明书.docx`

### 5. 生成源代码

本环节 **基于最新的Markdown格式的系统设计说明书文件（即最终确认版本）** 生成源代码。

调用技能 `code-from-design-doc` 来完成源代码的生成，流程如下：

1. **前置检查**：确认系统设计说明文档存在且符合要求
   - 文件格式：必须是 `.md` 格式
   - 文件名格式：`[软件名称]_系统设计说明书.md`
   - 文件版本：必须是用户确认后的最新版本

2. **调用 code-from-design-doc 技能**
   - 使用 Skill 工具调用 `code-from-design-doc` 技能
   - **传入Markdown格式的系统设计说明文档路径**作为参数（文件名格式：`<cwd>/[软件名称]_系统设计说明书.md`）
   - 技能将自动分析设计文档并生成完整的源代码（包括后端和前端）
   - 输出目录格式：`<cwd>/[软件名称]_源代码/`

3. **等待技能执行完成与检查**
    - 检查技能是否执行完成
    - 检查源代码输出目录是否存在，如果不存在请反馈用户并等待用户指令

### 6. 生成源代码提取文件

从第5步生成的源代码目录中提取纯代码内容：

1. **调用 code-extractor 技能**
   - 使用 Skill 工具调用 `code-extractor` 技能
   - 传入源代码目录路径（`<cwd>/[软件名称]_源代码/`）作为参数
   - 技能将自动对源代码进行纯代码提取
   - 输出文档：`code_extraction.md`

 2. **等待技能执行完成与检查**
    - 检查技能是否执行完成
    - 检查技能是否生成了代码提取文档，如果未生成请反馈用户并等待用户指令

## 二、文档生成规范

### 系统设计说明书结构

必须严格按照以下结构生成系统设计说明书：

1. **一、引言**
   - 1.1 系统概述
   - 1.2 开发目标
   - 1.3 开发时间
   - 1.4 运行环境
   - 1.5 目标读者（描述本系统面向的使用人群）
   - 1.6 参考资料（罗列1-3项参考资料）

2. **二、软件总体设计**
   - 2.1 架构设计（含Mermaid架构图）
   - 2.2 核心功能概述

3. **三、详细功能设计**
   - 各模块的功能描述、业务流程图或者时序图（Mermaid语法绘制）、图说明、输入输出

4. **四、技术实现说明**
   - 4.1 核心技术选型（后端、前端技术栈）
   - 4.2 关键技术点实现逻辑

5. **五、运行测试说明**
   - 5.1 测试环境
   - 5.2 测试用例

6. **六、软件功能列表**

7. **七、数据库设计**
   - 7.1 表结构设计（详细的表结构定义）
   - 7.2 实体关系图（ER图，使用Mermaid）

8. **八、接口文档**
   - 8.1 接口列表（按模块组织的RESTful API文档，包含路径、描述、参数、响应示例）
   - 8.2 接口调用示例（展示完整的请求和响应示例）
   - 8.3 通用错误码说明

### Mermaid 图表使用规范

在生成系统设计说明书时，应根据内容需要使用以下类型的 Mermaid 图表：

| 图表类型 | 使用场景 | 示例 |
|---------|---------|------|
| flowchart | 业务流程、操作流程 | 用户注册流程、订单处理流程 |
| sequence | 时序图、交互流程 | 前后端交互、API调用时序 |
| classDiagram | 类结构、类关系 | 后端模型类设计 |
| erDiagram | 实体关系图、数据库设计 | 用户-订单-商品关系 |
| graph | 架构图、拓扑图 | 系统总体架构 |

### .docx 文档生成规范

生成 Word 文档时，遵循以下规范：

1. **目录要求**
   - 文档开头必须包含目录
   - 目录格式：
     ```
     目录
     一、引言 ....................................... 1
       1.1 系统概述 ............................... 1
       1.2 开发目的 ............................... 1
       ...
     二、软件总体设计 ............................... 3
       2.1 架构设计 ............................. 3
       2.2 核心功能概述 ......................... 4
       ...
     ```
   - **包含三级标题到目录**
   - 使用中文数字（一、二、三...）标识一级标题
   - 使用页码显示章节位置

2. **格式要求**
   - 标题使用 Word 内置样式（标题1、标题2、标题3）
   - 正文字体：宋体，小四号（12磅）
   - **标题颜色统一为黑色**
   - 标题字体：黑体，加粗
   - 一级标题：黑体，16磅
   - 二级标题：黑体，14磅
   - 三级标题：黑体，13磅
   - 行距：1.5倍

3. **Markdown格式清理**
   - **去除四级及以上标题的#标记**（如 #### 8.3.1 获取订单列表 → 8.3.1 获取订单列表）
   - **去除粗体标记**（**路径参数** → 路径参数）
   - 去除斜体标记（*text*）
   - 去除代码标记（`text`）
   - 去除链接标记（[text](url) → text）

4. **图片插入**
   - 所有 Mermaid 图表转换为 PNG 后插入
   - PNG图片保存到 `<cwd>/assets/` 目录
   - 图片命名格式：`[软件名称]_图表[序号].png`
   - 图片居中对齐
   - 图片下方添加图表标题
   - 图片宽度不超过页面宽度（建议6英寸）
   - **支持重试机制，确保所有图表成功转换**

5. **代码块处理**
   - 代码块保留原始格式
   - 使用等宽字体（Consolas）
   - 添加浅灰色背景（#F5F5F5）

6. **表格处理**
   - 表格使用Word表格功能生成
   - 表格样式：Light Grid Accent 1
   - 表头加粗，居中对齐
   - 单元格字体：宋体，10.5磅
   - **自动清理表格内容中的Markdown标记**

7. **页码**
   - 文档包含页码
   - 页码居中显示在页面底部

## 三、脚本使用说明

### convert_mermaid_to_png.py

**功能**：将Markdown中的Mermaid代码块转换为PNG图片

**使用方法**：
```bash
python3 .claude/skills/software-copyright-applicant/scripts/convert_mermaid_to_png.py <md_file> <assets_dir> <software_name>
```

**参数说明**：
- `md_file`: Markdown文件路径
- `assets_dir`: 资源目录路径（用于保存PNG图片）
- `software_name`: 软件名称（用于图片命名）

**特性**：内置重试机制（3次）、失败时生成占位图片

### generate_docx.py

**功能**：生成含目录的Word文档

**使用方法**：
```bash
python3 .claude/skills/software-copyright-applicant/scripts/generate_docx.py <md_file> <docx_file> <assets_dir> --software-name <软件名称>
```

**参数说明**：
- `md_file`: Markdown文件路径
- `docx_file`: 输出的Word文档路径
- `assets_dir`: 资源目录路径（用于查找和保存图片）
- `--software-name`: 软件名称（用于转换Mermaid图表）

## 四、错误处理机制

执行各步骤时遇到错误，按以下原则处理：

- **非致命错误**（如Mermaid转换失败）：记录错误、生成占位、继续流程
- **致命错误**（如Word生成失败、技能调用失败）：中止流程，使用 AskUserQuestion 询问用户是否重试
- **所有错误**：向用户提供清晰的错误信息和原因

## 五、注意事项

1. **技术合理性**：技术选型必须符合实际业务场景和规模
2. **完整性**：文档必须包含软著申请所需的所有要素
3. **可读性**：文档内容清晰易懂，结构合理
4. **assets目录**：所有生成的静态资源（PNG图片）必须保存到 `<cwd>/assets` 目录
5. **目录生成**：.docx文档必须包含目录，使用中文数字标识一级标题
6. **Mermaid转换**：确保所有Mermaid图表成功转换为PNG图片
